# Агенты — Руководство пользователя

## Что такое агент?

Агент — это визуальный пайплайн обработки данных, построенный из нод (узлов), соединённых между собой. Каждая нода выполняет определённую функцию: вызов LLM, работа с памятью, логические операции, HTTP-запросы и т.д.

Агенты позволяют создавать сложные сценарии без написания кода — просто соединяя блоки на канвасе.

## Интерфейс редактора

### Основные элементы

- **Левая панель (Sidebar)** — библиотека доступных нод, сгруппированных по категориям
- **Канвас** — рабочая область для размещения и соединения нод
- **Панель свойств (Properties)** — настройки выбранной ноды
- **Панель запуска (Run Panel)** — результаты выполнения агента
- **Toolbar** — кнопки Run, Save, Export/Import

### Управление

- **Добавить ноду**: перетащите из левой панели на канвас
- **Соединить ноды**: потяните от выходного порта (●) к входному порту другой ноды
- **Удалить соединение**: выделите линию (клик) → нажмите Delete/Backspace
- **Удалить ноду**: ПКМ на ноде → Delete Node
- **Дублировать ноду**: ПКМ на ноде → Duplicate Node
- **Настроить ноду**: кликните на ноду → панель свойств справа

## Типы нод и их использование

### 🔹 Input / Output

**Input** — точка входа пайплайна. Может запрашивать данные у пользователя при запуске или использовать статическое значение.

**Output** — финальная нода. Показывает результат выполнения агента.

**Пример**: `Input → LLM → Output` — простейший чат-бот.

---

### 🧠 LLM

Отправляет промпт в Google Gemini и возвращает ответ модели.

**Настройки**:
- **Модель** — выбор модели (gemini-2.0-flash-exp, gemini-1.5-pro и т.д.)
- **API Ключ** — какой ключ использовать (если несколько)
- **Температура** — креативность (0 = детерминированно, 2 = очень креативно)
- **Макс. токены** — лимит длины ответа

**Входы**:
- `input` — основной текст для модели
- `prompt` — системный промпт (опционально)
- `context` — дополнительный контекст (опционально)

**Выходы**:
- `output` — ответ модели
- `error` — ошибка (если произошла)

**Пример**: Подключите `Input → LLM → Output` для базового Q&A.

---

### 📜 История чата (chat_history)

**Концепция**: Хранилище сообщений, привязанное к текущему чату. Каждый стор имеет свой `storeId` (например, `"dialog"`, `"game_history"`). Разные агенты могут использовать разные сторы.

**Операции**:
- **📖 Читать** — получить сообщения из стора
- **✏️ Добавить сообщение** — записать новое сообщение
- **📖✏️ Читать + добавить** — сначала прочитать старую историю, потом добавить новое (полезно для LLM)
- **🌱 Инициализировать (seed)** — заполнить стор начальными сообщениями
- **🗑️ Очистить** — удалить все сообщения

**Настройки**:
- **ID хранилища** — уникальное имя стора (например: `main`, `game_history`, `context`)
- **Операция** — что делать с историей
- **Роль для записи** — `user`, `assistant` или `system`
- **Фильтр по роли** — показывать только сообщения определённой роли
- **Лимит сообщений** — последние N сообщений (0 = все)
- **Формат текста** — как форматировать выход `text`:
  - `С ролями` — "User: ...\nAssistant: ..."
  - `Только текст` — без ролей
  - `XML` — `<user>...</user><assistant>...</assistant>`

**Входы**:
- `message` — текст сообщения для записи (при операции append)
- `seed_messages` — массив начальных сообщений (при операции seed)

**Выходы**:
- `messages` — массив объектов ChatHistoryMessage
- `text` — история в виде plain text (готово для подачи в LLM)
- `gemini_format` — массив `{role, parts}` для прямой подачи в Gemini API
- `count` — количество сообщений в сторе

**Примеры использования**:

#### Простой чат с историей
```
Input → chat_history(read_and_append, storeId:"dialog") → LLM → chat_history(append, role:assistant) → Output
```

1. Пользователь вводит вопрос
2. `chat_history` читает старую историю и добавляет новый вопрос
3. LLM получает всю историю как контекст
4. Ответ LLM сохраняется в историю как `assistant`
5. Результат показывается пользователю

#### РПГ-мастер с отдельными сторами
```
Input → chat_history(read, storeId:"game_state") → LLM → Split:
  → chat_history(append, storeId:"narrative")
  → chat_history(append, storeId:"game_state")
  → Output
```

Два независимых стора: один для нарратива, другой для игрового состояния.

---

### 💾 Память (memory_read / memory_write)

**memory_read** — поиск в долговременной памяти (глобальной или локальной).

**memory_write** — сохранение данных в память.

**Отличие от chat_history**: Память — это key-value хранилище с семантическим поиском. История чата — это упорядоченный список сообщений.

**Пример**: Сохранить факт о пользователе → позже найти его по запросу.

---

### ❓ Логика

#### Condition
Маршрутизация по условию. Напишите JavaScript выражение, которое возвращает `true`/`false`. Данные попадут либо в `True`, либо в `False` выход.

**Пример**: `input.length > 10` — если текст длиннее 10 символов, идёт в True.

#### Router
Маршрутизация по нескольким условиям (A, B, C, default).

**Режимы**:
- **Цепочка If-Else** — проверяет условия по порядку
- **Совпадение Regex** — проверяет input на соответствие regex
- **JS-выражение** — выполняет JS код для каждого маршрута

#### Loop
Batch-режим: обрабатывает все элементы массива последовательно.

**Выходы**:
- `current_item` — первый элемент
- `all_items` — все элементы массива
- `done` — завершён ли цикл

**Примечание**: Настоящая итерация (с повторным выполнением подграфа) не реализована из-за ограничений топологической сортировки.

#### Merge
Объединяет несколько потоков данных в один.

**Режимы**:
- **Склеить текст** — конкатенация строк с разделителем
- **Слить объекты** — Object.assign
- **Собрать в массив** — [a, b, c]
- **Первое непустое** — возвращает первый не-null

#### Split
Разделяет данные на части.

**Режимы**:
- По переносу строки
- По запятой
- По regex
- JSON массив

---

### 🛠 Утилиты

#### Transform
Преобразование данных через JS-выражение.

**Пример**: `return input.toUpperCase();`

#### Code
Выполнение произвольного JavaScript кода.

**Пример**:
```js
const result = input * 2;
return result;
```

#### HTTP Request
HTTP запрос к внешнему API.

**Настройки**: метод (GET/POST/PUT/DELETE), авторизация (Bearer, API Key, Basic Auth).

#### Debug
Pass-through нода с логированием в консоль. Полезна для отладки.

#### Text
Статический текст. Полезно для промптов и констант.

## Примеры агентов

### 1. Простой Q&A агент
```
Input → LLM → Output
```

Базовый чат-бот без памяти.

---

### 2. Агент с историей диалога
```
Input → chat_history(read_and_append, storeId:"dialog", role:user)
      ↓
     LLM (получает историю как контекст)
      ↓
     chat_history(append, storeId:"dialog", role:assistant)
      ↓
    Output
```

Полноценный чат с сохранением истории. LLM видит весь предыдущий диалог.

---

### 3. Агент-классификатор
```
Input → LLM (classify) → Router (regex_match):
  → route_a: "positive" → Output "😊"
  → route_b: "negative" → Output "😞"
  → default → Output "🤔"
```

Классифицирует текст и маршрутизирует по категориям.

---

### 4. RAG (Retrieval-Augmented Generation)
```
Input → memory_read (semantic_search) → Merge (с Input) → LLM → Output
```

Ищет релевантную информацию в памяти и подаёт её в LLM как контекст.

---

### 5. Мультишаговый исследователь
```
Input → LLM (generate questions) → Split (by newline) → Loop → LLM (answer each) → Merge → memory_write → Output
```

Генерирует вопросы, отвечает на каждый, сохраняет результаты в память.

## FAQ / Troubleshooting

### Агент не запускается
- Проверьте, что все обязательные входы подключены (красные порты)
- Убедитесь, что есть хотя бы одна Input и одна Output нода
- Проверьте API ключ в настройках LLM ноды

### Соединение не создаётся
- Типы портов должны быть совместимы (или один из них `any`)
- Нельзя соединить ноду саму с собой
- Проверьте, что тянете от выхода (справа) ко входу (слева)

### История чата не сохраняется
- Убедитесь, что `storeId` одинаковый во всех нодах chat_history
- Проверьте, что операция установлена в `append` или `read_and_append`
- История хранится в localStorage — очистка браузера удалит её

### LLM возвращает ошибку
- Проверьте API ключ в Sidebar → Settings
- Убедитесь, что модель доступна (некоторые модели требуют whitelist)
- Проверьте лимиты токенов — слишком длинный промпт может вызвать ошибку

### Как очистить историю чата?
Добавьте ноду `chat_history` с операцией `🗑️ Очистить` и запустите агента.

### Можно ли использовать несколько историй в одном агенте?
Да! Используйте разные `storeId` для разных контекстов. Например:
- `storeId: "dialog"` — основной диалог
- `storeId: "system_logs"` — системные логи
- `storeId: "game_state"` — состояние игры

## Советы по дизайну агентов

1. **Начинайте с простого** — Input → LLM → Output, потом добавляйте сложность
2. **Используйте Debug ноды** — для отладки промежуточных результатов
3. **Именуйте ноды осмысленно** — кликните на ноду и измените label в свойствах
4. **Сохраняйте часто** — кнопка "Save Agent" в правом верхнем углу
5. **Экспортируйте важные агенты** — Export → .json файл для бэкапа
6. **Тестируйте по частям** — используйте Output ноды для проверки промежуточных результатов
