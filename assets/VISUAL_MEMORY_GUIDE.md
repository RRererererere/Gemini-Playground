# Визуальная память — Руководство

## Обзор

Визуальная память — это интеллектуальная система для работы с изображениями в Gemini Studio. Модель сама решает когда искать изображения и использует их в контексте разговора.

## Ключевые возможности

### 1. Самостоятельный поиск модели

Модель имеет доступ к функциям `search_image_memories()` и `recall_image_memory()` и сама решает когда их использовать.

**Как это работает:**
```
Пользователь: "Покажи фото Маши"
  ↓
Модель видит в prompt: "У тебя есть search_image_memories()"
  ↓
Модель вызывает: search_image_memories("Маша")
  ↓
Получает результаты: [{id: "abc", entities: ["Маша"], description: "...", mentions: 5}]
  ↓
Модель анализирует и решает: нужно ли recall
  ↓
Модель вызывает: recall_image_memory("abc")
  ↓
Получает изображение и отвечает пользователю
```

**Преимущества:**
- Модель сама решает когда искать (не всегда нужно)
- Может искать несколько раз с разными query
- Видит метаданные (mentions, tags) и принимает решение
- Может найти связанные изображения через повторный search

### 2. Честная метрика mentions

Теперь `mentions` инкрементится только при реальном использовании:
- ✅ Когда модель вызывает `recall_image_memory()`
- ❌ НЕ когда изображение просто попало в prompt

Это позволяет точно отслеживать, какие изображения действительно используются.

### 3. Детальный просмотр изображений

Новый компонент `ImageMemoryDetailModal` показывает:
- Полноразмерное изображение
- Описание, теги, entities (редактируемые)
- Метаданные (дата, количество использований, scope)
- Связанные изображения (crops, похожие)
- Связанные текстовые воспоминания
- Возможность редактирования и удаления

### 4. Улучшенный поиск в Memory Modal

Два отдельных поля поиска:
- Поиск текстовой памяти (по факту, keywords)
- Поиск изображений (по description, tags, entities)

Клик по изображению открывает детальный просмотр.

### 5. Визуальные индикаторы в чате

**ImageMemoryRecallPill** — показывает когда модель вспомнила изображение:
```
👁️ Вспомнено изображение: Маша
```

**ImageMemoryPill** — показывает когда модель сохранила изображение:
```
💾 Сохранено в глобальную память: Маша
```

### Поток данных

```
User Message: "Покажи фото Маши"
    ↓
buildMemoryPrompt() → добавляет инструкции
    ↓
Model sees: "У тебя есть search_image_memories()"
    ↓
Model calls: search_image_memories("Маша")
    ↓
searchImageMemories() → [{id, description, tags, entities, mentions}]
    ↓
Model analyzes results
    ↓
Model calls: recall_image_memory("abc")
    ↓
incrementImageMemoryMentions() → mentions++
    ↓
UI: ImageMemoryRecallPill
    ↓
Model receives image and responds
```

## Confidence Score

Confidence вычисляется на основе:
1. **Количество результатов** (0-0.5): `min(count/3, 1) * 0.5`
2. **Релевантность** (0-0.3): `avgScore * 0.3`
3. **Прямые упоминания** (0-0.2): `(directMentions/totalEntities) * 0.2`

Пороги:
- `> 0.7` — высокий (зелёный)
- `> 0.4` — средний (жёлтый)
- `≤ 0.4` — низкий (серый)

## Извлечение entities

Паттерны:
- Имена с заглавной буквы: `Маша`, `Nike`, `Иван Петров`
- Конструкции "это X": `это моя подруга`, `это логотип`
- Вопросы: `кто такой X`, `что такое Y`

Фильтрация:
- Минимум 3 символа
- Исключены служебные слова: `это`, `про`, `такой`, `кто`, `что`

## Примеры использования

### Сценарий 1: Поиск человека

```
User: "Покажи фото Маши"
  → Model: search_image_memories("Маша")
  → Response: [{id: "abc", entities: ["Маша"], description: "Девушка Маша...", mentions: 5}]
  → Model: recall_image_memory("abc")
  → UI: "👁️ Вспомнено изображение: Маша"
  → Model: [показывает изображение пользователю]
```

### Сценарий 2: Поиск с уточнением

```
User: "Что я тебе рассказывал про Машу?"
  → Model: search_image_memories("Маша")
  → Response: [{id: "abc", entities: ["Маша"], description: "...", mentions: 5}, ...]
  → Model анализирует: есть 3 изображения, mentions показывает что часто обсуждали
  → Model: "У меня есть 3 фото Маши. Она твоя подруга, вы часто обсуждали..."
  → (может вызвать recall если нужно показать конкретное фото)
```

### Сценарий 3: Множественный поиск

```
User: "Покажи всех моих друзей"
  → Model: search_image_memories("друзья")
  → Response: [{entities: ["Маша"]}, {entities: ["Петя"]}, ...]
  → Model: search_image_memories("Маша")
  → Model: search_image_memories("Петя")
  → Model собирает информацию и отвечает
```

### Сценарий 3: Сохранение нового изображения

```
User: [прикрепляет фото] "Это моя подруга Катя"
  → Model: save_image_memory(
      image_id="ph_abc123",
      description="Девушка Катя, подруга пользователя",
      tags=["person", "female", "friend"],
      entities=["Катя"],
      scope="global"
    )
  → UI: "💾 Сохранено в глобальную память: Катя"
  → Memory Modal: новое изображение появляется в списке
```

## Лучшие практики

### Для пользователей

1. **Называйте людей и объекты по имени** при загрузке фото
   - ✅ "Это моя подруга Маша"
   - ❌ "Это фото"

2. **Используйте Memory Modal** для управления
   - Просмотр всех сохранённых изображений
   - Редактирование описаний и тегов
   - Удаление ненужных

3. **Проверяйте preflight результаты**
   - Если система нашла не то — уточните запрос
   - Если не нашла — проверьте что изображение сохранено

### Для разработчиков

1. **Не блокируйте preflight**
   - Он выполняется быстро (синхронно)
   - Не требует API вызовов

2. **Используйте confidence для UI**
   - `> 0.6` — автоматически добавлять в контекст
   - `> 0.3` — показывать в UI
   - `≤ 0.3` — не показывать

3. **Тестируйте edge cases**
   - Пустые entities
   - Дубликаты изображений
   - Большое количество результатов

## Производительность

- **Preflight lookup**: ~5-10ms (синхронный)
- **searchImageMemories**: ~2-5ms для 100 изображений
- **Thumbnail size**: ~3KB (80x80px)
- **localStorage limit**: ~5MB для индекса (~1500 изображений)
- **IndexedDB**: неограниченно для полных изображений
