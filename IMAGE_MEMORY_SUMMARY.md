# Image Memory System — Краткое резюме

## 🎯 Что это решает

Gemini Studio сейчас **забывает изображения** после закрытия чата. Пользователь кидает фото подруги Маши — Gemini отвечает, но при следующем упоминании не помнит как она выглядит.

Image Memory System добавляет **визуальную долгосрочную память** — Gemini может запоминать людей, места, объекты, аннотации, и использовать их в будущих разговорах и при создании сайтов.

## 🔑 Ключевые решения

### 1. Автоматическая классификация (Type A vs Type B)

**Gemini сам решает** что сохранять:

- **Type A (сохранять)**: люди с именами, места, логотипы, референсы
- **Type B (не сохранять)**: скриншоты "куда нажать", баги, временные задачи

Через промпт с чёткими критериями.

### 2. Дедупликация через двойной хэш

- **SHA-256** — точные дубликаты (одинаковые байты)
- **pHash** (перцептивный хэш) — похожие изображения
- **Hamming distance ≤10** — считаем похожими

Если фото уже есть — не создаём дубликат, а увеличиваем `mentions`.

### 3. Трёхуровневое хранение

```
localStorage (5-10MB)
  └─ image_memory_index — метаданные + thumbnails 80x80

IndexedDB (50-500MB)
  ├─ img_mem_full_${id} — полное изображение
  └─ img_mem_annotated_${id} — с аннотациями
```

Thumbnails всегда доступны мгновенно. Полные изображения загружаются по требованию.

### 4. Аннотации как часть памяти

Когда Gemini обводит лицо Маши через `annotate_regions` — аннотация **сохраняется вместе с изображением**. В следующем чате он вспомнит не только фото, но и что именно было обведено.

### 5. Кропы как отдельные воспоминания

Групповое фото → Gemini обводит каждое лицо → сохраняет:
- Полное фото (ImageMemory #A)
- Кроп лица Маши (ImageMemory #B, `sourceImageMemoryId: #A`)
- Кроп лица Пети (ImageMemory #C, `sourceImageMemoryId: #A`)

Каждый кроп — отдельное воспоминание с тегами и описанием.

### 6. Website Builder через mem:// протокол

Gemini генерирует HTML:

```html
<img src="mem://img_mem_abc123" alt="Маша" />
```

При рендере в iframe — `mem://` резолвится в `data:image/jpeg;base64,...`

**Преимущества:**
- Чистый читаемый HTML
- Не раздувает код base64
- При экспорте — отдельная функция подставляет base64

### 7. Поиск без векторной БД

Keyword matching + tag overlap + entity match:

```typescript
score = tagOverlap * 2 + entityMatch * 3 + descOverlap + mentions * 0.1
```

Для 50-200 изображений — достаточно. При упоминании "Маша" — тег даёт точное совпадение.

## 🎨 Как это выглядит для пользователя

### Сценарий 1: Запоминание человека

```
Пользователь: [отправляет фото] Это моя подруга Маша

Gemini: 
  [вызывает annotate_regions — обводит лицо]
  [вызывает save_image_memory — сохраняет в память]
  Приятно познакомиться с Машей! Запомнил её фото.

--- Новый чат на следующий день ---

Пользователь: Помнишь мою подругу?

Gemini:
  [вызывает search_image_memories("подруга")]
  [находит Машу]
  [вызывает recall_image_memory]
  Да, это Маша! Вот её фото [показывает]. Тёмные волосы, улыбается.
```

```
Пользователь: Создай сайт-визитку для Маши

  [находит фото]
  [генерирует HTML с <img src="mem://img_mem_abc123">]
  
Пользователь видит:
  - Сайт с фото Маши из памяти
  - Чистый HTML код без огромных base64 строк
```

### Сценарий 3: Групповое фото

```

  - Полное фото (ID: abc123)

Пользователь: Покажи фото Маши

  [вызывает search_image_memories("Маша")]

```

## 🔧 Технические детали

### Размеры и квоты

- **Thumbnail**: 80x80 JPEG quality 0.8 ≈ 3KB
- **Полное изображение**: 200KB-2MB (зависит от разрешения)
- **localStorage лимит**: 5-10MB → ~50 thumbnails + мета
- **IndexedDB лимит**: 50-500MB → ~100-200 полных изображений

### Производительность

- **pHash вычисление**: ~50-100ms на изображение
- **SHA-256 вычисление**: ~10-20ms
- **Thumbnail generation**: ~20-30ms
- **Поиск по 100 записям**: <5ms

### Обратная совместимость

- Не ломает существующую текстовую память
- Новая таблица `image_memory_index` в localStorage
- Новые ключи в IndexedDB с префиксом `img_mem_`
- Старые чаты продолжают работать без изменений
